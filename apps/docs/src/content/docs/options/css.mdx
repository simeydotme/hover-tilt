`HoverTilt` wraps your content with two elements:

- A **`container`** element that handles perspective and listens for pointer events.
- An inner **`tilt`** element that applies transforms and contains the glare effect.

## Styling the internals

Here's a look at the internal structure of the rendered component;

<Tabs syncKey="api">
  <TabItem label="Svelte">

    ```txt
    <div class="hover-tilt-container">
      └── <div class="hover-tilt">
            └── slotted content
    ```

    In Svelte, you can style the component's internal elements via `class` on the `<HoverTilt>`, and then build a css declaration, or apply TailwindCSS classes directly to the component.

    <Tabs syncKey="css">
      <TabItem label="Plain CSS">
        ```html
        <HoverTilt class="my-component">
          <div class="content-area">
            Your content here
          </div>
        </HoverTilt>
        ```
        ```css
        .my-component {
          border-radius: 24px;
          .hover-tilt {
            opacity: 0.5;
          }
        }
        .content-area {
          border-radius: 24px;
        }
        ```
      </TabItem>
      <TabItem label="TailwindCSS">
        ```html
        <HoverTilt class="rounded-3xl [&_.hover-tilt]:opacity-50">
          <div class="rounded-3xl">
            Your content here
          </div>
        </HoverTilt>
        ```
      </TabItem>
    </Tabs>

  </TabItem>
  <TabItem label="Web Component">
    ```txt
    <hover-tilt>
      ├── #shadow-root
      │     └── <div part="container">
      │           └── <div part="tilt">
      │                 └── <slot />
      └── slotted content (Light DOM)
    ```

    So for internal styling on the web-component,
    [use `::part(container)` and `::part(tilt)` CSS selectors](https://developer.mozilla.org/en-US/docs/Web/CSS/Reference/Selectors/::part) to style the shadow DOM elements.

    ### `::part(container)`

    > wraps the tilt effect.

    _Generally you don't want to modify the style of this,
    except for things like `perspective` or `perspective-origin`._

    ### `::part(tilt)`

    > is the animated layer that holds the glare pseudo-element.

    _`::part(tilt)` inherits the `border-radius` of the container, so you can
    either apply a radius to the `::part(container)` or the `::part(tilt)`
    (see below for more details)._

    <Tabs syncKey="css">
      <TabItem label="Plain CSS">
        ```html
        <hover-tilt class="my-component">
          <div class="content-area">
            Your content here
          </div>
        </hover-tilt>
        ```
        ```css
        .my-component::part(container) {
          border-radius: 24px;
        }
        .my-component::part(tilt) {
          opacity: 0.5;
        }
        .content-area {
          border-radius: 24px;
        }
        ```
      </TabItem>
      <TabItem label="TailwindCSS">
        ```html
        <hover-tilt class="[&::part(container)]:rounded-3xl [&::part(tilt)]:opacity-50">
          <div class="rounded-3xl">
            Your content here
          </div>
        </hover-tilt>
        ```
      </TabItem>
    </Tabs>

  </TabItem>
</Tabs>

## CSS custom properties

`<HoverTilt />` mirrors its internal properties into CSS variables so you can style content,
gradients, or additional layers without writing extra JavaScript. These values are emitted on the
container element (or `::part(container)` in the web component).

<Aside type="tip">
  When used as a **Web Component**, the variables are [exposed via `::slotted(*)`](https://developer.mozilla.org/en-US/docs/Web/CSS/::slotted) so that they can be picked up by your slotted content.
</Aside>

## Other considerations

There's a couple of other considerations to keep in mind when styling the component, due
to the way it is structured and how web-components work.

### `border-radius` on glare

If you are applying a `border-radius` to the content, you may notice that the
glare overflows the radius. This is because the glare is applied to the `::before` pseudo-element of the `.hover-tilt` element.

So you typically want the container to match border radii (or any clipping) on
that inner layer to avoid the glare spilling past rounded corners. (see example below)

<br /> <br />

<Tabs syncKey="api">

  <TabItem label="Svelte">
    The Svelte component exposes a `class` prop on the container. And the 
    `tilt` and `::before` elements both inherit the `border-radius` of the container.

    So as long as we apply the same `border-radius` to the container as the content,
    the glare will not overflow the radius.

    <RadiusSvelte />

    <Code lang="svelte" title="Card.svelte" code={svelteExample} />

  </TabItem>

  <TabItem label="Web Component">
    The custom element renders inside Shadow DOM, so classes on `<hover-tilt>` 
    only affect the host and not the internal container/tilt nodes. 
    
    Use the [CSS `::part()` selector](https://developer.mozilla.org/en-US/docs/Web/CSS/::part) to 
    reach those internal elements.

    <RadiusWebC />

### `overflow` on container

Because the tilt element is inside of the container, if you apply an `overflow: hidden`
to the container, the tilt effect will be cut off and look broken.

If you need to clip content to the card area, then apply an `overflow: hidden`
to the tilt element, or more preferably to the slotted content.

### `perspective` of the tilt effect

The `<HoverTilt>` component has a default `perspective` of `600px`.

If you need to change the perspective, you can apply it to the container with CSS;

### content over the glare

You may notice that the glare effect sits on top of the slotted content, this means that if you have text inside the slotted content, it will be covered by the glare.

While this is phsyically correct, it may not be visually desirable. So it's possible to move the slotted content up in the z-order to sit atop the glare.

<div class="not-content mx-8 grid gap-4 md:grid-cols-2">
  <hover-tilt>
    <div class="demo-card min-w-0! from-amber-400! to-rose-400!">
      <h3 class="rounded-md bg-slate-500 p-2 px-4 text-slate-50 shadow-md text-shadow-none">Behind Glare</h3>
    </div>
  </hover-tilt>
  <hover-tilt>
    <div class="demo-card min-w-0! from-amber-400! to-rose-400!">
      <h3 class="z-10 rounded-md bg-slate-500 p-2 px-4 text-slate-50 shadow-md text-shadow-none">Over Glare</h3>
    </div>
  </hover-tilt>
</div>

All we need to do is apply a `z-index` to the slotted content to move it up in the z-order.

### `z-index` adjacent components

If you are using the `scaleFactor` prop, you may find that when the component zooms in, it overlaps with other components nearby. This will be especially noticeable in a gallery of scaling cards;

<div class="not-content mx-8 grid gap-4 md:grid-cols-2">
  <hover-tilt scale-factor={1.3}>
    <div
      class="demo-card aspect-square! min-w-0!"
      style="background: linear-gradient(135deg, #667eea 0%, #764ba2 100%);"
    >
      <h3>Basic Example</h3>
      <p>Hover over this card to see the default tilt effect</p>
    </div>
  </hover-tilt>
  <hover-tilt scale-factor={1.3}>
    <div
      class="demo-card aspect-square! min-w-0!"
      style="background: linear-gradient(135deg, #667eea 0%, #764ba2 100%);"
    >
      <h3>Basic Example</h3>
      <p>Hover over this card to see the default tilt effect</p>
    </div>
  </hover-tilt>
</div>

But it is easily resolved by applying a `z-index` to the `<HoverTilt>` component when it is being hovered;

<Tabs syncKey="api">
  <TabItem label="Svelte">
    <Tabs syncKey="css">
      <TabItem label="Plain CSS">
        ```svelte
        <HoverTilt scaleFactor={1.3} class="my-component">
          Your content here
        </HoverTilt>
        ```
        ```css
        .my-component:hover {
          z-index: 10;
        }
        ```
      </TabItem>
      <TabItem label="TailwindCSS">
        ```svelte
        <HoverTilt scaleFactor={1.3} class="hover:z-10">
          Your content here
        </HoverTilt>
        ```
      </TabItem>
    </Tabs>
  </TabItem>
  <TabItem label="Web Component">
    <Tabs syncKey="css">
      <TabItem label="Plain CSS">
        ```html
        <hover-tilt scale-factor={1.3} class="my-component">
          Your content here
        </hover-tilt>
        ```
        ```css
        .my-component:hover {
          z-index: 10;
        }
        ```
      </TabItem>
      <TabItem label="TailwindCSS">
        ```html
        <hover-tilt scale-factor={1.3} class="hover:z-10">
          Your content here
        </hover-tilt>
        ```
      </TabItem>
    </Tabs>
  </TabItem>

This is not a part of the component's internals because every use case is different and it would have been frustrating for users to battle with a `:hover` selector, and `z-index` value they didn't want.

Making it something you have to handle manually means you can use different, more complex logic instead of just `:hover {}` selectors (if you desire).
